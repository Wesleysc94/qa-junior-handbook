## O problema que o POM resolve

Sem Page Object, o teste vira isso: `page.locator('.inventory_item').nth(3)` espalhado em cinco arquivos. Quando a UI muda uma classe, você gasta o dia inteiro caçando cópia. Com **Page Object Model**, você centraliza **locators** e **ações** (ex.: `addToCartByName`) numa classe ou módulo — o spec só descreve **comportamento**.

No SauceDemo, os fluxos são curtos, mas a repetição aparece rápido: todo teste que compra algo passa por **login** e **inventário**. Sem POM você duplica; com POM você reutiliza e reduz custo de manutenção — métrica que entrevistador técnico entende na hora.

## LoginPage — porta de entrada

Responsabilidades típicas: preencher usuário e senha, clicar em login, **opcionalmente** expor asserção de erro de credencial. Locators preferíveis: `getByRole('textbox', { name: ... })` ou label, não XPath frágil.

**Erro comum:** colocar **assertion pesada** dentro da Page (“esperar que URL seja X e que fulano apareça”). O ideal é a Page expor **estado acionável** (`isErrorVisible()`), e o **teste** decidir o `expect`. Misturar demais deixa Page difícil de reutilizar em fluxos negativos.

```ts
// Ilustração — adapte aos seus seletores reais
// loginPage.login('standard_user', 'secret_sauce');
```

## InventoryPage — catálogo e seleção

Aqui mora a maior parte dos locators de produto: cards, botões “Add to cart”, ordenação, menu. Métodos como `addFirstAvailableToCart()` ou `addByProductName('Sauce Labs Backpack')` encapsulam política de seleção.

**Por que nomear por produto quando possível?** Porque falha de teste com “nth(2)” não diz nada no relatório; “não achei Sauce Labs Bike Light” diz.

## CartPage — estado intermediário

Carrinho é **estado**: você valida quantidade, remoção, continuar comprando. Page Object deve expor ações curtas: `open()`, `removeItem()`, `checkout()`. Se o carrinho usa badge numérico no header, decida se o locator fica na `InventoryPage` ou numa `HeaderComponent` compartilhada — **composição** vence herança gigante.

## CheckoutPage — formulário e validação

Checkout em SauceDemo costuma ser um ou dois passos (informações + overview). Separe **preencher campos** de **submeter**, para reutilizar em teste negativo (“deixar ZIP vazio”). Assertions de erro de campo são ótimos candidatos a **soft assert** se você quiser listar tudo de uma vez — mas não abuse: para fluxo crítico, falha rápida pode ser melhor.

## CompletePage — ponto de confirmação

A página de “Thank you” é âncora de sucesso: `expect(page.getByText('Thank you'))` ou equivalente. Centralizar `orderCompleteMessage()` evita duplicar texto mágico em dez arquivos.

## Como os testes consomem as Pages

Padrão legível: `test('compra feliz', async ({ page }) => { const login = new LoginPage(page); ... })` ou fixtures customizadas que já entregam `loggedInPage`. O segundo reduz ruído quando metade da suíte começa logada — mas esconder demais o login pode confundir leitor do teste; equilíbrio é documentar no README.

## Como evoluiria se o produto crescesse

Com mais telas: extrair **componentes** (Header, Footer), introduzir **facades** para fluxos (“comprar produto X com frete Y”), e avaliar **API mocking** para isolar E2E de indisponibilidade externa. Para SauceDemo isso é overkill — mas na entrevista mostra visão de **escala**.
