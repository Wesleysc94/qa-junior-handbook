## Por que dois workflows separados

Em monorepos, é comum **api-tests.yml** e **e2e-tests.yml** (nomes ilustrativos) com gatilhos `paths:` diferentes. Motivo: mudança só em documentação não precisa baixar browser; mudança só em teste de API não precisa instalar Playwright completo se você segmentar — **economiza minutos** e reduz fila no GitHub Actions.

Trade-off: mais YAML para manter. Mitigação: comentários no topo de cada workflow (“dispara quando muda pasta X”) e template reutilizável se o time crescer.

## Anatomia mínima que o recrutador espera ver

- **on:** `push` e `pull_request` (às vezes `workflow_dispatch` para rerun manual).
- **permissions:** restritos ao necessário (`contents: read` é padrão seguro em muitos casos).
- **jobs:** um para API (Python: setup-python, cache pip/poetry) e outro para E2E (setup-node, `npx playwright install --with-deps` no Linux).
- **steps:** checkout, instalar deps, rodar testes, **upload de artefato** em falha.

## REQRES_API_KEY e variáveis de ambiente

Mesmo APIs públicas às vezes exigem **chave** em exemplos avançados ou em forks privados. O padrão é: valor em **GitHub Secret**, leitura no workflow via `${{ secrets.REQRES_API_KEY }}`, e **nunca** logar o valor. No pytest, `os.environ["REQRES_API_KEY"]` ou arquivo `.env` só local (listado no `.gitignore`).

Erro comum de júnior: commitar `.env` com chave — repositório público vira vazamento em minutos. Se isso acontecer em estudo, **rote a chave** e apague histórico com ferramenta adequada; na entrevista, conte o que aprendeu.

## Artefatos que salvam debug

Quando E2E falha no CI, você não tem seu monitor: precisa de **trace**, **screenshot**, **vídeo** ou relatório HTML. `actions/upload-artifact@v4` com `if: failure()` ou `if: always()` garante que o recrutador (ou você daqui a três meses) baixa a pasta e reproduz.

Configure Playwright para gerar trace **on-first-retry** ou **on-failure** conforme política — equilíbrio entre tamanho de artefato e riqueza de debug.

## O que evoluir no pipeline

- **Matriz de browsers** (`chromium`, `firefox`, `webkit`) em job paralelo — aumenta confiança e custo de minutos.
- **Sharding** para suíte grande: `npx playwright test --shard=1/3` etc.
- **Reuso com `workflow_call`** se repetir etapas entre projetos.
